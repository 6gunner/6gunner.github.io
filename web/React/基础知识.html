    # 组件的生命周期

    简单图示

    复杂的图示

    我们把react的生命周期分为三个阶段：

    # 挂载阶段

    ==挂载时==react组件会经历以下三个生命周期函数。 在render以后，react会去更新dom和ref引用。

    constructor -> static getDerivedStateFromProps -> render -> componentDidMount.

    老版本的react还有UNSAFE_componentWillMount()生命周期函数

    # 更新阶段

    当react组件传入的props发生变化时，或者当react组件setState方法被调用时，组件进入更新的阶段。

    ==更新时==react组件经过以下三个阶段

    static getDerivedStateFromProps -> shouldComponentUpdate -> render -> getSnapshotBeforeUpdate -> componentDidUpdate

    老版本的react还会触发UNSAFE_componentWillReceiveProps，UNSAFE_componentWillUpdate()生命周期函数

    state更新时日志

    所以实际上只要父级重新渲染时，这两个生命周期函数就会重新调用，不管 props 有没有“变化”

    # 卸载阶段

    ==卸载==的时候，组件会触发componentWillUnmount函数

    # 生命周期函数的意义

    # componentDidMount()

    组件挂载完成后调用，此时组件已经插入 DOM 树中了。

    依赖于 DOM 节点的初始化应该放在这里。

    # componentDidUpdate()

    componentDidUpdate(prevProps, prevState, snapshot) 会在==组件更新后立即调用==，但是首次渲染不会被执行。

    可以在这里进行props的判断，从而根据props的变化来进行网络请求。

    componentDidUpdate(prevProps) {
  // 典型用法（不要忘记比较 props）：
  if (this.props.userID !== prevProps.userID) {
    this.fetchData(this.props.userID);
  }
}

    也可以在 componentDidUpdate() 中直接调用 setState()，但请注意它必须被包裹在一个条件语句里，要注意比较prevProps和当前props，否则会死循环。

    # componentWillUnmount()

    componentWillUnmount()会在组件卸载时被调用，在此方法里执行必要的清理操作。比如一些定时任务、一些订阅等。

    # shouldComponentUpdate()

    shouldComponentUpdate(nextProps, nextState)的返回值，react组件会去做为是否重新渲染的依据。

    默认会返回true，即每次props或者state发生变化时，都会重新渲染。

    如果返回false， React 则跳过更新，不会调用UNSAFE_componentWillUpdate、render()、componentDidUpdate()。

    但是返回 false ==并不会阻止子组件在 state 更改时重新渲染==。

    ==首次渲染或使用 forceUpdate()== 时不会调用该方法。

    以下是一些不常用的生命周期函数

    # getDerivedStateFromProps 派生属性

    // todo 详细阅读这篇文章 https://zh-hans.reactjs.org/blog/2018/06/07/you-probably-dont-need-derived-state.html

    getDerivedStateFromProps 的存在只有一个目的：让组件在 props 变化时更新 state。

    之前经常用的是componentWillReceiveProps()。

    所以可能的使用场景有两个：

    • 无条件的根据 prop 来更新内部 state，也就是只要有传入 prop 值， 就更新 state
    • 只有 prop 值和 state 值不同时才更新 state 值。

    在 React 16.4^ 的版本中 setState 和 forceUpdate 也会触发这个生命周期，所以内部 state 变化后，又会走 getDerivedStateFromProps 方法，并把 state 值更新为传入的 prop。

    以下是过期的生命周期函数

    # UNSAFE_componentWillReceiveProps()

    UNSAFE_componentWillReceiveProps(nextProps) 会在已挂载的组件接收新的 props 之前被调用.

    ==this.setState不会触发这个生命周期函数，只会在pros更新时，调用此方法==

    你可以比较 this.props 和 nextProps 并在此方法中使用 this.setState() 执行 state 转换。(常见用法)。

    请注意

    ==如果父组件导致组件重新渲染，即使 props 没有更改，也会调用此方法。== （可以参考上面更新阶段-图2）

    如果只想处理props更改时的逻辑，请确保进行当前值与变更值的比较。

    # UNSAFE_componentWillUpdate()

    当组件收到新的 props 或 state 时，会在渲染之前调用 UNSAFE_componentWillUpdate()。

    ==初始渲染不会调用此方法。==

    在componentWillUpdate()返回前，==不能执行任何操作来触发react组件的更新==。

    ==比如不能在这个方法里调用this.setState()，否则会造成死循环==

    此方法可以替换为 componentDidUpdate()。

    如果你在此方法中读取 DOM 信息（例如，为了保存滚动位置），则可以将此逻辑移至 getSnapshotBeforeUpdate() 中。

    如果 shouldComponentUpdate() 返回 false，则不会调用 UNSAFE_componentWillUpdate()。

    # UNSAFE_componentWillMount()

    componentWillMount()只有在挂载阶段才被调用，

    它在render之前被调用，在这个方法里调用setState不会触发额外渲染。

    # 受控组件、非受控组件

    之前听说过受控、非受控的概念是用在表单的input组件上的。 如果一个表单的输入框，它的值是通过外部传入value来进行修改的，那就是受控的；反之则是非受控组件。

    在组件里，如果一个组件的数据可以通过props来传入的话，则组件是受控组件。如果组件的数据只能维护在内部的state中，则是非受控组件。

    尽量避免使用props复制到state的用法。

    原因一、如果直接复制 prop 到 state，会导致父组件重新渲染时，子组件的state被覆盖会原始值。

    class EmailInput extends Component {
  state = { email: this.props.email };

  render() {
    return <input onChange={this.handleChange} value={this.state.email} />;
  }

  handleChange = event => {
    this.setState({ email: event.target.value });
  };

  componentWillReceiveProps(nextProps) {
    // 这会覆盖所有组件内的 state 更新！
    // 不要这样做。
    this.setState({ email: nextProps.email });
  }
}

    原因二、会导致props不变化时，组件的state不会变化，而存在共享输入值的问题。

    class EmailInput extends Component {
  state = {
    email: this.props.email
  };

  componentWillReceiveProps(nextProps) {
    // 只要 props.email 改变，就改变 state
    if (nextProps.email !== this.props.email) {
      this.setState({
        email: nextProps.email
      });
    }
  }
  // 这里正常情况下，是不会有问题。
  // 但想象一下，如果这是一个密码输入组件，拥有同样 email 的两个账户进行切换时，这个输入框不会重置。（设计层面的问题）
}

    # Ref的使用方式总结

    之前版本的react有两种ref的写法

    字符串形式、回调函数形式。

    字符串形式是已经被废弃了

    回调函数的形式写法如下：

    <Component ref={ref => this.comRef = ref}></Component>

    没错，他是传入一个回调函数，然后将ref赋值给comRef变量。

    # 新版本写法

    新版本ref常见的是两种使用方式：React.createRef, React.useRef。

    相同点

    两者都可以创建一个mutable对象，用来保存临时的对象，这个对象可以是dom引用，也可以是普通的js对象。

    区别

    useRef是react的钩子函数，只能用在Function Component里。

    createRef一般只用在组件函数里。原因是：

    • createRef每次执行时，不会对上一次的ref值进行保留。如果createRef用在function component里，每次组件重新渲染时，都会被重新执行。导致ref值丢失，取不到ref.current的值。
    • useRef只在组件初始化（第一次被调用）时被赋予初始值，之后组件重新渲染调用该方法，值依然保持不变。

    所以createRef一般都在React Component的初始化生命周期里执行。比如：constrcutor、componentWillMount、

    useRef在Function Component里使用，你可以把它的作用理解为：将第n帧的props或者state保存在ref的current对象上，然后在第n+1帧里去读取ref的值。

    useRef的使用方法

    function App {
    const divRef = React.useRef() // 存储dom
    const valueRef = React.useRef(90) // 存储普通number
    const [,setDummyState] = useState()
    return (
        <div>
            Value: {valueRef.current}
            <div id="divR" ref={divRef}>App, here</div>
            <button onClick={()=> (valueRef.current = 88, setDummyState({}))}>Incr</button>
        </div>
    )
}

    利用useRef来避免闭包问题

    function Timer() {
	const [count, setCount] = useState(0);
	const ref = useRef();
	ref.current = count; // 每次count变化时，都会重新给ref赋值
	const handleClick = () => {
		setTimeout(() => {
			setCount(ref.current + 1);
		}, 1000);
	}
	return (
		<div>
			<p>{count}</p>
			<button onClick={() => setCount(count + 1)}>
				setCount
      </button>
			<button onClick={handleClick}>
				delay setCount
      </button>
		</div>
	)
}

    # 常见问题

    # 2.Can't perform a React state update on an unmounted component

    现象如标题所说，不能在unmounted的组件上进行state的更新操作。

    这个现象发生的原因一般有2种，一个是忘了取消定时任务，一个是异步请求回来的时候，触发了setState，但是这个时候组件已经销毁了。

    解决方案：

    第一种很好解决，在componentWillUnmount里面取消定时任务

    第二种，因为一般异步任务不好去手动取消，所以一般是在setState的时候，对component的状态进行判断。

    可以写一个装饰器(没验证过)

    function withUnmounted (target){
  // 改装componentWillUnmount，销毁的时候记录一下
  let next = target.prototype.componentWillUnmount
  target.prototype.componentWillUnmount = function () {
    if (next) next.call(this, ...arguments);
    this.unmount = true
  }
  // 对setState的改装，setState查看目前是否已经销毁
  let setState = target.prototype.setState
  target.prototype.setState = function () {
    if ( this.unmount ) return ;
    setState.call(this, ...arguments)
  }
}
@withUnmounted
class BaseComponent extends Component {
}

    或者手动去修改component的组件

    class Page extends Component {
  _isMounted = false;

  state = {
    isLoading: true

    this._isMounted = true;

  componentWillUnmount() {

  }
